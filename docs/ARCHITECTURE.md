# Architecture

## Data model

Five JSON files generated by Faker live in `data/`:

| File | Records | Key |
|------|---------|-----|
| `stores.json`      | 500   | `store_id`           |
| `vendors.json`     | 1000  | `vendor_account_id`  |
| `employees.json`   | ~120  | `id`                 |
| `departments.json` | 7     | `name`               |
| `admins.json`      | 7     | `username` (bcrypt-hashed password lives here) |

Each store carries its address, phone, email, opening date, store manager,
district manager (`reports_to → VP`), and VP of Operations. Each store has
exactly two vendor accounts: phone and internet, each with provider,
account number, monthly cost, support phone, portal URL and a synthetic
login (`username` + `password`).

## Retrieval pipeline

```
question  ─►  retriever (FAISS, top-k semantic similarity)
                      │
                      ▼
              top-k passages (LangChain Documents)
                      │
                      ▼
        ChatPromptTemplate(system + user)
                      │
                      ▼
                ChatOllama(llama3.2:3b)
                      │
                      ▼
                  StrOutputParser
                      │
                      ▼
                   answer (str)
```

The chain is composed with LCEL inside `backend/rag_engine.py`:

```python
self.chain = (
    {"context": retriever | _format_docs, "question": RunnablePassthrough()}
    | prompt
    | self.llm
    | StrOutputParser()
)
```

Sources are surfaced separately from the LLM call — the front-end shows
the user exactly which passages fed the answer.

## Voice pipeline

```
[ user clicks mic ] ─► MediaRecorder (browser, audio/webm)
                              │
                              ▼
                   POST /api/transcribe (multipart)
                              │
                              ▼
            HuggingFace transformers.pipeline(
                "automatic-speech-recognition",
                model="openai/whisper-tiny.en"
            )
                              │
                              ▼
                  text -> chat input box
                              │
                              ▼
                       /api/chat (RAG)
                              │
                              ▼
        SpeechSynthesisUtterance reads the answer aloud
                  (browser-native, no install)
```

The voice dependencies (`transformers` + `torch`) are split into a separate
`requirements-voice.txt`. If they are not present, `/api/transcribe`
returns 501 with a helpful message and the rest of the app keeps working.
TTS uses the browser's built-in `SpeechSynthesis` API to keep the install
small and deployment cross-platform.

## Why these choices

- **`llama3.2:3b` via Ollama**: small, free, runs on a laptop, good enough
  for structured-lookup style questions where the answer is in context.
- **`all-MiniLM-L6-v2`**: 384-dim embeddings, ~80 MB, fast on CPU, the
  workhorse default for English text retrieval.
- **`whisper-tiny.en`**: ~75 MB, ~2 s transcription on CPU, and it pulls
  cleanly through the `transformers` pipeline that the IBM course
  emphasises.
- **Browser-side TTS**: zero extra install for the user, and crucially
  it does not compete with the LLM and embedder for CPU.
- **FAISS local**: zero infra, the whole index fits in RAM, rebuilds in
  seconds after an admin edit.
- **Flask + vanilla JS**: matches the IBM course content, keeps the
  surface area small for franchise owners who want to fork it.

## Admin update flow

```
admin signs in (email + password)
        │
        ▼
   /api/login  ─►  bcrypt verify  ─►  JWT (8 h)
        │
        ▼
   /admin form: dataset + id_field + id + JSON patch
        │
        ▼
   /api/admin/update  ─►  patch JSON file  ─►  RAGEngine.rebuild_index()
        │
        ▼
   chatbot now answers from the updated data
```

## Roadmap (good "next steps" for a course capstone)

- **Server-side TTS** with a HuggingFace model such as
  `microsoft/speecht5_tts` or `facebook/mms-tts-eng` — useful if the
  deployment target is a kiosk or call-center where the browser cannot
  speak.
- **Per-department write scopes** (e.g. Procurement can patch vendors
  but not employees).
- **Audit log** of every admin update.
- **Streaming LLM tokens** to the front-end with Server-Sent Events.
- **Swap Llama 3.2:3b for IBM `granite-3.0`** via watsonx for a closer
  fit to the IBM course narrative.
- **Multilingual mode** — switch the Whisper model to `whisper-tiny`
  (no `.en`) and adjust the system prompt to detect Spanish.
